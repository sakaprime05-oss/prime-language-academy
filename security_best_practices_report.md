# Security Best Practices Report

## Executive summary

The application is in a stronger security posture after this pass: high-severity dependency findings were removed, upload access is restricted to internal roles, public appointment input is validated and escaped before email rendering, the admin bot payment path no longer creates arbitrary payment plans, and baseline HTTP security headers are configured.

The main remaining risk is operational rather than a direct code exploit: invoice PDFs are generated by sending invoice data to a third-party service. This should be replaced later with local PDF generation if payment privacy needs to be tighter.

## Fixed findings

### SEC-001 - High - Vulnerable Next.js dependency

- Location: `package.json`
- Evidence: `npm audit --audit-level=high` initially reported high-severity advisories against `next@16.1.6`.
- Impact: public Next.js apps can inherit framework-level request handling or DoS vulnerabilities.
- Fix applied: upgraded Next.js to `^16.2.4`; `npm audit --audit-level=high` now exits successfully.

### SEC-002 - Medium - Upload endpoint allowed every authenticated user

- Location: `src/app/api/upload/route.ts:26`
- Evidence: upload now checks `session.user.role !== "ADMIN" && session.user.role !== "TEACHER"`.
- Impact: before the fix, any logged-in student could upload public PDF/image files through the shared upload endpoint.
- Fix applied: uploads are now restricted to admin and teacher roles.

### SEC-003 - Medium - Public appointment fields were inserted into HTML emails

- Location: `src/app/api/public-appointment/route.ts:6`, `src/app/api/public-appointment/route.ts:26`, `src/app/api/public-appointment/route.ts:86`
- Evidence: added `escapeHtml`, field length limits, email validation, and date/time validation before rendering HTML.
- Impact: attacker-controlled appointment fields could create unsafe HTML in emails sent to admin or visitors.
- Fix applied: public appointment fields are normalized, limited, validated, and escaped.

### SEC-004 - Medium - Admin bot payment validation was too permissive

- Location: `src/app/api/admin-bot/route.ts:7`, `src/app/api/admin-bot/route.ts:16`, `src/app/api/admin-bot/route.ts:206`
- Evidence: added timing-safe API key comparison and moved payment validation to `validatePaymentFromBot`.
- Impact: if the bot key were misused, the old code could create a default payment plan and validate arbitrary manual amounts.
- Fix applied: bot payment validation now requires an existing payment plan and rejects amounts above the remaining balance.

### SEC-005 - Low - Missing baseline security headers

- Location: `next.config.ts:4`
- Evidence: configured `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy`.
- Impact: lack of basic headers leaves the app with weaker browser-side protection against clickjacking, MIME sniffing, and excessive browser permissions.
- Fix applied: baseline headers now apply to all routes.

### SEC-006 - Low - LLM model initialized at module load

- Location: `src/app/actions/ai-grader.ts:5`, `src/app/actions/ai-grader.ts:23`, `src/app/actions/ai-grader.ts:59`
- Evidence: Gemini model is now initialized lazily, input is capped, and the returned score is clamped.
- Impact: eager initialization is less build-safe and unbounded prompt input can increase cost and reliability risk.
- Fix applied: lazy model getter, transcript/question length caps, max score clamp.

## Remaining findings

### SEC-007 - Medium - Invoice generation sends payment data to a third-party service

- Location: `src/app/api/invoice/[id]/route.ts:53`
- Evidence: invoice data is posted to `https://invoice-generator.com`.
- Impact: names, emails, amounts, and payment references leave the application boundary. This may be acceptable temporarily, but it is a privacy and dependency risk.
- Recommended fix: generate PDFs locally with a server-side library or a controlled Vercel function, then avoid sending invoice data to an unrelated external service.

### SEC-008 - Low - Remaining moderate dependency advisories

- Location: `package-lock.json`
- Evidence: `npm audit --audit-level=high` now passes, but `npm audit` still reports moderate issues involving `postcss` under Next and `uuid` through Resend/Svix.
- Impact: not currently high severity, and the suggested forced fixes are breaking or odd for this dependency graph.
- Recommended fix: monitor package updates and update Next/Resend again when non-breaking patched versions are available.

## LLM recommendation

A small LLM can help the system, but it should start as a read-only assistant:

- admin morning summary: registrations, payments to verify, appointments, Club capacity
- student placement-test scoring and short feedback
- email draft improvement for admin approval, not automatic sending
- support FAQ assistant using only public PLA program data

Avoid giving an LLM direct payment validation, user status changes, or automatic email-send authority until audit logs and approval screens are in place.
